# mycflow 開発計画

[01 基本概念](01%20mycflow%20基本概念.md)・[02 機能設計](02%20mycflow%20機能設計.md) を前提とした段階的な開発計画。
「初期スコープは固定DAG + wac + 起動時Zenoh ACLで十分。動的グラフは自己修復・再配置フェーズから」という機能設計の方針（02の3節）に沿ってフェーズを区切る。

## フェーズ一覧

| フェーズ | 名称 | ゴール | 状態 |
|---|---|---|---|
| 0 | リポジトリ基盤 | モノレポ雛形・CI 4段の骨格 | **完了**（2026-07-07） |
| 1 | WITインターフェース確定 | `mycflow:dataflow` / `mycflow:node` の初版凍結 | 進行中（現フェーズ） |
| 2 | 固定DAG縦通し（Docker） | compose環境でsensor→aggregatorのE2Eが通る | 未着手 |
| 3 | データフロー強制 | 起動時Zenoh ACL + wac合成による三層強制の実証 | 未着手 |
| 4 | 自律配置 | 負荷・通信量に基づく配置判断と再配置 | 未着手 |
| 5 | 自己修復・動的グラフ | ノード障害からの自動再配置、ホスト関数層の許可リスト機構 | 未着手 |
| 6 | Kubernetes展開 | containerd + runwasi / RuntimeClassでの運用 | 未着手 |
| 7 | エッジ・実機 | KubeEdge、WAMR on Zephyr/ESP-IDF | 未着手 |

各フェーズは前フェーズの成果物の上に積むが、フェーズ1と2は一部並行できる（スタブ実装で縦通し環境を先に作り、WIT変更を流し込む進め方）。

---

## フェーズ0: リポジトリ基盤（完了）

モノレポ構成・Rustワークスペース・WIT雛形・compose定義・CI 4段（WIT検証 → cargo test → コンポーネントビルド → compose E2E）を整備済み。
経緯は [ADR 0001](adr/0001-monorepo-structure.md)（モノレポ採用）・[ADR 0002](adr/0002-exclude-atomvm-otp.md)（AtomVM除外）を参照。

## フェーズ1: WITインターフェース確定（現フェーズ）

**ゴール**: `wit/` のインターフェース初版を凍結し、以降の破壊的変更をバージョン管理下に置く。

| 項目 | 内容 |
|---|---|
| 成果物 | `mycflow:dataflow@0.1.0` / `mycflow:node@0.1.0` / `mycflow:worlds@0.1.0` の確定版 |
| 主な作業 | channel resourceの送受信セマンティクス確定（下記の要決定事項）、`open-channel` 許可チェックフックのシグネチャ確定、lifecycle報告項目の確定 |
| 完了条件 | `wasm-tools component wit wit/` が通り、フェーズ2の両コンポーネントがこのWITだけでbindgenできる |

**このフェーズで決めること**（02の5節「未決事項」より）
- モジュール間通信の同期/非同期モデル（現状の `receive` はブロッキング想定。非同期化するならpollable/stream導入をここで判断）→ 決定をADR化
- エラー型の粒度（`dataflow-error` のvariantが運用に足りるか）
- `resource-usage` 報告の項目・単位（配置判断の入力になるため、フェーズ4の要求から逆算して過不足を確認）

## フェーズ2: 固定DAG縦通し（Docker）

**ゴール**: `docker compose up` だけで sensor-sim → aggregator の固定DAGがZenoh経由で流れ、E2Eテストが自動判定する。

| 項目 | 内容 |
|---|---|
| 成果物 | 実装済み `components/sensor-sim` `components/aggregator`（cargo-component + wasm32-wasip2）、wasmtime統合を持つ `node-agent`、`tests/e2e/run.sh` |
| 主な作業 | cargo-component導入とbindgen、`host-dataflow` のZenohバックエンド実装（`AllowAll` チェッカーのまま）、node-agentへのwasmtime組み込み、ローカルOCIレジストリからのwkg pull |
| 完了条件 | CIのcompose E2E段が実データで合否判定する（現状の echo TODO を置き換え） |

このフェーズでは強制機構は入れない（全許可）。「動くパイプライン」を最短で作り、以降のフェーズの検証土台にする。

## フェーズ3: データフロー強制

**ゴール**: 02の3節の三層強制（静的・動的・真正性）のうち、固定DAGスコープの静的+動的を実証する。

| 項目 | 内容 |
|---|---|
| 成果物 | `compositions/*.wac` の実体化（wac合成バイナリ）、placement-managerのトポロジ→Zenoh ACL生成（`config/zenoh/acl-template.json5` の実体化）、etcdへのトポロジ投入 |
| 主な作業 | wacによる事前結線、起動時ACL書き込みフロー、`PermissionChecker` のetcdトポロジ参照版実装 |
| 完了条件 | トポロジ定義に**ない**チャネルへの `open-channel` が `permission-denied` で拒否されるE2Eテスト（負のテスト）が通る |

真正性（OCIダイジェスト署名とpub/sub権限の紐付け）は設計だけ行い、実装はフェーズ6（実運用レジストリが登場する段階）へ送る。

## フェーズ4: 自律配置

**ゴール**: 配置マネージャーが判断材料（CPU/メモリ負荷・データ通信量・モジュール転送コスト）に基づいてモジュールの配置先を決定・変更する。

| 項目 | 内容 |
|---|---|
| 成果物 | placement-managerの配置判断ロジック、node-agentのメトリクス報告（`mycflow:node/lifecycle` 経由）、「停止→転送→別ノードで起動」の再配置フロー |
| 主な作業 | 配置コスト関数の設計（通信量 vs モジュール転送量の比較）、etcd配置メタデータの更新プロトコル、データ局所性を考慮した近接配置 |
| 完了条件 | compose環境で負荷の偏りを人工的に作り、モジュールが効率の良いノードへ自動移動するデモが動く |

**前提となる決定**: モジュールのステートレス化と外部ストア委譲の詳細（02の5節）。再配置はステートレス前提のため、このフェーズの冒頭で状態委譲の規約を決めてADR化する。

## フェーズ5: 自己修復・動的グラフ

**ゴール**: ノード障害の検知と健全ノードへの自動再配置。トポロジの実行時変更に対応する。

| 項目 | 内容 |
|---|---|
| 成果物 | 障害検知（Zenoh liveliness API + etcdリース）、自動再配置、ホスト関数層の許可リスト機構（インスタンス単位の許可キー式リストを呼び出しごとに検証） |
| 主な作業 | Zenoh ACLの実行時更新不可という制約への対策として、強制をホスト関数層へ移す（02の3節「動的強制・動的グラフの場合」）。Zenoh ACLは多層防御の外壁として残す |
| 完了条件 | ノードコンテナをkillしてもDAGが健全ノード上で自動復旧し、その間の許可リストが再起動なしで更新される |

**前提となる決定**: 固定DAG/動的グラフの方針確定（02の5節）。フェーズ3〜4の運用経験を踏まえてこのフェーズ冒頭で判断し、ADR化する。**配置マネージャー自体の冗長化設計**（単一障害点の回避）もこのフェーズで扱う。

## フェーズ6: Kubernetes展開

**ゴール**: compose模擬環境から containerd + runwasi / Kubernetes へ移行する（Kube-Wasmパターン）。

| 項目 | 内容 |
|---|---|
| 成果物 | `deploy/k8s/` の実体化（RuntimeClass、runwasi設定、マニフェスト一式）、K8sスケジューラと配置マネージャーの役割分担の確定、モジュール真正性（署名検証）の実装 |
| 完了条件 | kind等のローカルクラスタで `runtimeClassName: wasm` のPodとしてDAGが稼働する |

## フェーズ7: エッジ・実機

**ゴール**: KubeEdge（CloudCore/EdgeCore）による間欠接続対応と、WAMR on Zephyr/ESP-IDF での実機実行。

| 項目 | 内容 |
|---|---|
| 成果物 | `deploy/kubeedge/` の実体化、WAMR統合（`host-dataflow` のC API境界実装）、実機用ファームウェア |
| 備考 | Zephyr/ESP-IDFのビルドシステム（west, idf.py）はCargoワークスペースと共存が難しいため、このフェーズで `mycflow-firmware` リポジトリとして分離する（[ADR 0001](adr/0001-monorepo-structure.md) 参照）。`wit/` の独立リポジトリ化もインターフェース凍結状況を見てこのタイミングで判断 |

## 未決事項とフェーズの対応

02の5節の未決事項は、以下のフェーズで決定しADRに記録する。

| 未決事項 | 決定フェーズ |
|---|---|
| WITインターフェースの具体的設計 | フェーズ1 |
| 同期/非同期通信モデルの最終決定 | フェーズ1 |
| モジュールのステートレス化・外部ストア委譲の詳細 | フェーズ4冒頭 |
| 固定DAG/動的グラフの方針確定 | フェーズ5冒頭 |
| 配置マネージャーの冗長化設計 | フェーズ5 |

## 運用ルール

- フェーズをまたぐ設計判断は必ず `docs/adr/` に記録する（[テンプレート](adr/0000-template.md)）。
- 各フェーズの完了条件はCIで自動判定できる形（E2Eテスト・デモスクリプト）に落とす。
- WITの破壊的変更はフェーズ1完了（0.1.0凍結）以降、パッケージバージョンを上げて行う。
