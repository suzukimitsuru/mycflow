# mycflow 機能設計

## アーキテクチャ全体（層構成）

| 層 | 技術 | 責務 |
|---|---|---|
| 組み込みデバイス | WAMR（WebAssembly Micro Runtime）on Zephyr / ESP-IDF | モジュールのロード・実行・リソース報告のみ。配置判断は持たない |
| コンテナ | containerd + runwasi（Bytecode Alliance製containerd shim） | Kubernetesで `runtimeClassName: wasm` によりスケジューリング・監視 |
| エッジサイト | KubeEdge（CloudCore / EdgeCore） | CloudCoreがスケジューリング決定、EdgeCoreが間欠接続下のローカル自律性を維持 |
| 通信 | Zenoh | 組み込みデバイスとコンテナ制御プレーン間の軽量プロトコル |
| パッケージ配布 | wkg + OCIレジストリ | WASMコンポーネントの一元配布（全層で互換性確認済み） |
| 状態管理 | etcd/Consul（配置メタデータ）+ CRDT/イベントソーシング（業務データ） | 強整合性/結果整合性の2層分離 |

kubeletが`runtimeClassName: wasm`のPodをスケジューリングすると、containerdがrunwasi経由でWasmtime/WasmEdge/WAMR（組み込み・エッジ向け）のいずれかにルーティングする。これによりK8sのスケジューラ・ヘルスチェック・メトリクス収集が配置と監視を担い、デバイス側のWAMRは純粋な実行係に留まる。

## 1. 配置マネージャー
- 判断材料: CPU/メモリ負荷、データ通信量、モジュール転送コスト、ノード障害検知
- データフローのトポロジグラフをetcd上の一次情報として保持し、配置決定とデータフロー許可（後述）を同一ソースから導出する
- **未確定**: 配置マネージャー自体の冗長化設計（単一障害点になりうる）

## 2. パッケージ管理（wkg / OCIレジストリ）
- 参照形式: `namespace:package@version`（例: `mycflow:sensor-driver@1.0.0`）
- 名前空間→レジストリのマッピングは `wasm-pkg/config.toml` で設定し、論理名と物理格納場所を分離
- 実体は専用メディアタイプ（`application/vnd.wasm.config.v0+json`）を持つOCIアーティファクト。Dockerイメージと同じOCI Distribution API上で運用
- `wkg.toml`（プロジェクト単位・任意の依存上書き）/ `wkg.lock`（言語をまたいで共通のロックファイル）
- containerd+runwasiはOCIレジストリからのpullを前提に動くため、Kubernetesのスケジューリング決定に自然に連動
- 組み込み側（WAMR）も薄いエージェント経由で同じOCI Distribution APIからpullする、サーバ・組み込み共通のパイプライン

## 3. データフロー強制（WASMコンポーネント間）
WASMサンドボックスは「宣言外のシステムリソースに触れない」ことは保証するが、「どのモジュールと通信してよいか」は別問題。これを3層で強制する。

### 静的強制（ノード内）
- WITのworld定義でimport/exportを明示し、宣言外のやり取りを構造上不可能にする
- `wac`（WebAssembly Composition）で事前にコンポーネントを結線し、固定グラフ以外の経路をバイナリ上に存在させない

### 動的強制（ノード間・固定DAGの場合）
- 配置マネージャーがetcd上の固定グラフ定義に従い、各ノードのZenoh設定ファイルにACLルールを**起動時**に書き込んで有効化する

### 動的強制（ノード間・実行時にトポロジが変わる動的グラフの場合）
- **制約**: Zenoh ACLは設定ファイルベースで実行時更新不可（反映にはインスタンス再起動が必要）。大規模で動的な環境への拡張性に課題があり、トークン認可の提案も未実装のオープンイシュー
- **対策**: 強制をネットワーク層（Zenoh ACL）からホスト関数層に移す
  - コンポーネントはZenohに直接触れず、WITで定義したカスタムpub/subインターフェース（例: `mycflow:dataflow/channel`）経由でのみ通信する
  - ホスト側（WAMR/wasmtime統合層）が、配置マネージャー配布の「インスタンス単位の許可キー式リスト」を呼び出しごとに検証する
  - リストは配置マネージャーがetcdグラフの変更を検知して都度更新可能。Zenohノード自体の再起動は不要
  - Zenoh ACLは多層防御の外側の壁（インフラ侵害対策）として残す
- Zenoh liveliness API（ノードがキー式で稼働を宣言し、他ノードが検知）で動的グラフの「誰が今稼働しているか」を補う（許可そのものは保持しない）
- **方針**: 初期スコープは固定DAG + wac + 起動時Zenoh ACLで十分。動的グラフは自己修復・再配置フェーズから必要になるため、今はWITインターフェース設計の段階で「許可チェックのフック点」だけ用意しておく

### モジュールの真正性
- OCI配布されたWASMコンポーネントのダイジェスト・署名をZenoh上のpub/sub権限に紐付け、なりすましモジュールの不正経路流入を防ぐ

## 4. 検討済み代替案と判断

| 候補 | 結果 |
|---|---|
| wasmCloud（NATSラティス + wRPC） | 配置の自律性という核心コンセプトに近く、当初は最有力だったが、責務分離方針の確定後はcontainerd+runwasi中心の構成に収束 |
| Microsoft Orleans | Activation Rebalancing/Repartitioningが核心コンセプトに最も近いが、.NET/C#エコシステム前提 |
| Apache Pekko / Dapr / Ray | コンテナ層（サーバ/クラウド側のスケジューリング・配置ロジック）限定で採用可能な候補として保持 |
| Erlang / AtomVM | **除外**。OTP supervisionがランタイム内蔵のため、「実行はデバイス、監視はコンテナ層」という責務分離方針と矛盾 |

## 5. 未決事項（今後の検討課題）
- WASMコンポーネントモデル（WITインターフェース）の具体的設計
- モジュール間通信の同期/非同期モデルの最終決定（gRPC候補 / NATS候補）
- 配置マネージャー自体の冗長化設計
- モジュールのステートレス化と外部ストア委譲によるマイグレーションコスト削減の詳細化
- データフローが固定DAGか動的グラフかの方針確定（動的グラフの場合、ホスト関数層の許可リスト機構の実装が必要）
