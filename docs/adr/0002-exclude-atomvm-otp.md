# ADR 0002: Erlang/OTP・AtomVMを除外する

- ステータス: 採用（除外）
- 日付: 2026-07-07

## コンテキスト

mycflowの設計原則は「組み込みデバイスは実行のみ、スケジューリング・監視はコンテナ層に集約する」という責務分離である（[docs/01 mycflow 基本概念.md](../01%20mycflow%20基本概念.md) 3節参照）。
Erlang/OTP・AtomVMは自己修復・関数型思想がmycflowのコンセプトと一致し、MCUまで実行系を統一できる唯一の候補として一時有力視された。

## 決定

Erlang/OTP・AtomVMを実行ランタイム候補から除外し、WAMR（純粋なサンドボックス）を採用する。

## 理由

OTPのスーパービジョンツリー（監視機能）はランタイムに内蔵されている。これは「監視をコンテナ側に分離する」という責務分離方針と思想的に矛盾する。WASM/WAMRはもともと「ただのサンドボックス」であり配置判断や監視機能を持たないため、この役割分担に自然に収まる。

## 結果

- 実行ランタイムはWAMR（組み込み）+ containerd/runwasi（コンテナ）に統一する（[docs/01 mycflow 基本概念.md](../01%20mycflow%20基本概念.md) 4.1/4.2節）。
- モジュールはステートレスを前提とし、再配置は「停止→転送→別ノードで起動」で実現する方針と整合する。
- 他の除外候補（wasmCloud、Orleans、Pekko、Dapr、Ray、NATS）についても評価経緯を別途ADR化する余地がある（未着手）。
