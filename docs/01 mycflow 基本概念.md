# mycflow 基本概念

## 概要
- **プロジェクト名**: 菌糸フロー環境（Mycelium Flow Environment）
- **略称・ディレクトリ名**: `mycflow`（mycelium + flow の造語。検索識別性とディレクトリ名としての実用性を兼ねて採用）
- **インスピレーション**: 菌糸ネットワークの自己組織化
- **コアコンセプト**: WASMモジュールを自律的に配置・実行し、ノード障害時には自己修復する環境

## 「自律」の定義
判断主体は配置マネージャー（環境側）。

**判断材料**
- CPU/メモリ負荷
- モジュール間のデータ通信量
- モジュール転送コスト（バイナリの配置コスト）
- ノード障害の検知

**アクション**
- モジュールを別ノードへ移動・複製
- 障害ノードのモジュールを健全ノードへ再配置
- データ局所性（Data Locality）を意識した近接配置（データとモジュールを近づける）

## 責務分離の原則（最重要設計原則）
- **組み込みデバイス**: モジュールのロード・実行・リソース報告のみを担う、純粋な実行サンドボックス。スケジューリング・監視は一切持たない
- **コンテナ層**: スケジューリング・監視・配置判断をすべて集約
- この制約が、Erlang/AtomVMを候補から後退させる決定打となった（OTPのsupervisionがランタイム内蔵のため、分離設計と矛盾するため）

## スコープ
- **初期**: Dockerベースのマルチノード（VM）環境
- **長期目標**: 組み込みシステムからクラウドまでをスパンする分散実行環境

## 状態管理の方針
ステートレス徹底は現実的でないと判断し、分散ステートを許容。整合性要求の違いから2層に分離する。

| 層 | 内容 | 整合性 | 技術候補 |
|---|---|---|---|
| Layer 1 | 配置メタデータ（どのモジュールがどこにいるか、ノードの死活状態） | 強整合性 | etcd / Consul |
| Layer 2 | モジュールの業務データ本体 | 結果整合性 | CRDT / イベントソーシング |

将来的には、モジュール自体をステートレスに保ち状態を外部ストアへ委譲することで、再配置（マイグレーション）コストを下げる方針を検討中（詳細は未確定）。

## 評価の経緯
段階的に評価軸を広げてから絞り込んだ:
1. WASM/Rustを中心とした初期検討
2. 代替案の幅出し: Microsoft Orleans、Apache Pekko、Dapr、Ray、Erlang/OTP
3. 組み込み制約の追加 → AtomVM、Zenohを検討
4. 「実行環境はスケジューリング・監視を持たない」という制約の確定 → WAMR中心のアーキテクチャに収束
