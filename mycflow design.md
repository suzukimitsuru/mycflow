# mycflow（菌糸フロー環境）基本概念・機能設計

## 1. プロジェクト概要

| 項目 | 内容 |
|---|---|
| プロジェクト名 | 菌糸フロー環境 |
| 略称・ディレクトリ名 | `mycflow`（mycelium + flow） |
| コンセプト | 菌糸ネットワークの自己組織化からインスピレーションを得た、自律的なモジュール配置・実行環境 |
| 初期ターゲット | Dockerベースのマルチ仮想マシン環境 |
| 長期ターゲット | 組み込みデバイスからクラウドまでをスパンする分散実行環境 |

### 命名の経緯
- 「菌糸ネットワーク」は自律的配置のイメージを伝えるためのメタファーであり、設計指針そのものではなく命名に留める方針を確認済み。
- `mycelium` `hyphae` `rhizome` `wasmCloud` 等は既存プロジェクトと重複するため除外。
- 短さ・意味の明確さ・検索でのユニーク性を基準に `mycflow` を採用。次点は `anastom`（菌糸の接合・融合を表す専門用語）。

---

## 2. コアコンセプト：「自律性」の定義

自律の判断主体は **配置マネージャー（環境側）** であり、モジュール自身が判断するわけではない。

**判断材料**
1. CPU/メモリ負荷
2. データ通信量（モジュール間のデータ転送コスト）
3. モジュール転送コスト（バイナリの配置コスト）
4. ノード障害の検知

**アクション**
- 負荷分散・通信量とモジュール転送量を比較し、効率の良い方を優先してモジュールを配置
- データとモジュールを近接配置し転送コストを削減（データ局所性）
- ノード障害時に健全ノードへ自己修復・再配置

---

## 3. 設計原則：責務分離

mycflowの核心は **「組み込みデバイスは実行のみ、スケジューリング・監視はコンテナ層に集約する」** という責務分離である。

この制約が技術選定の最大の決定打となった。WASMはもともと「ただのサンドボックス」であり配置判断や監視機能を持たないため、この役割分担に自然に収まる。一方 Erlang/AtomVM は OTP のスーパービジョンツリー（監視機能）がランタイムに内蔵されているため、「監視をコンテナ側に分離する」という方針と思想的に矛盾し、不採用となった。

---

## 4. アーキテクチャ

### 4.1 組み込みデバイス層 — WAMR
- **WAMR（WebAssembly Micro Runtime）** を純粋な実行サンドボックスとして採用。
- Zephyr・ESP-IDFなど主要RTOS上で動作し、ホスト環境への組み込み用に薄いC APIを提供。「ロード・実行・リソース報告」のみを担い、配置判断ロジックは一切持たない。
- モジュールの再配置は、モジュールがステートレスである限り「停止→転送→別ノードで起動」で実現でき、状態の外部委譲方針と整合する。

### 4.2 コンテナ層 — containerd + runwasi
- **containerd + runwasi**（Bytecode AllianceがCNCFに寄贈したcontainerdシム）を採用。
- Kubernetesが `runtimeClassName: wasm` のPodをスケジューリングし、containerdがWasmtime/WasmEdge/WAMR（組み込み・エッジ向け）へルーティングする構成（Kube-Wasmパターン）。
- スケジューラ・ヘルスチェック・メトリクス収集はすべてコンテナ側のコントロールプレーンが担い、デバイス側のWAMRは純粋な実行係に留まる。

### 4.3 エッジサイト — KubeEdge
- 間欠接続環境への対応として **KubeEdge** を採用。
- クラウド側 **CloudCore** がスケジューリング決定を担当し、エッジ側 **EdgeCore** が接続断時もローカル自律性を維持する分離モデル。

### 4.4 通信層 — Zenoh
- 組み込みデバイス〜コンテナ制御プレーン間の軽量プロトコルとして **Zenoh** を採用。
- 最小実装は8bitマイコン上で約300バイトのフットプリント。ベアメタルからデータセンターまでブリッジ無しに単一プロトコルで通信可能。
- pub/sub・分散ストレージ・クエリ・計算を統一抽象化しており、「モジュール間通信」と「分散状態管理の一部」を同時に担える。
- Rust実装であり、WASM/Rust路線との親和性も高い。

### 4.5 モジュール配布 — wkg + OCIレジストリ
- **wkg**（Bytecode Alliance製, wasm-pkg-tools）でWASMコンポーネントとWITインターフェースをOCIレジストリ上のアーティファクトとして一元管理。
- `namespace:package@version` 形式（例: `mycflow:sensor-driver@1.0.0`）で参照し、物理的な格納場所は `wkg.toml` の名前空間マッピングが解決。
- 実体は専用メディアタイプを持つOCIアーティファクトであり、Dockerイメージと同じOCI Distribution API上で扱える。
- `wkg.lock` は言語に依存しない共通ロックファイル形式（Rust/Go等、どの言語で書いても同一）。
- containerd+runwasiはOCIレジストリからのpullを前提に動作するため、Kubernetes層の配置決定がそのままwkg管理下のコンポーネント取得につながる。組み込み機器への配布もOCIアーティファクトのpullとして同一パイプラインに乗る。

### 4.6 状態管理 — 二層分離設計
ステートレス徹底は構成上困難なため、分散ステートを許容。ただし整合性要求の異なるデータを分離する。

| レイヤー | 内容 | 採用技術 |
|---|---|---|
| Layer 1: 配置メタデータ | どのモジュールがどこにいるか、ノードの死活状態（強整合性が必要） | etcd / Consul |
| Layer 2: 業務データ | モジュールが処理するデータ本体（結果整合性で可） | CRDT / イベントソーシング |

この分離をしないと配置判断とデータ整合性が混在し複雑化するため、明確に層を分けている。

## 5. モジュール間データフローの制御

WASMのサンドボックスはリソース分離を保証するが、「どのコンポーネントがどのコンポーネントと通信できるか」は別途制御が必要という前提のもと、三層構成で強制する。

1. **静的強制（ノード内）**: WIT worldによるインターフェース定義 + `wac` によるコンポーネント合成
2. **動的強制（ノード間）**: Zenohのキー式ACL + etcd管理のトポロジグラフ
3. **真正性の保証**: OCIダイジェストに紐づく署名付きモジュールIDにより、宣言されたデータパスへの不正なコンポーネント注入を防止

配置マネージャーがトポロジグラフからZenoh ACLを生成する権限を持つことで、**配置決定とデータフロー許可が単一のソースオブトゥルースを共有する**設計になっている。

> 未決定: データフロートポロジが固定DAGか、実行時に変化する動的グラフかによって、ACL生成のタイミングと整合性保証の設計が変わる（要検討）。

## 6. 検討した代替案と評価の経緯

評価は段階的に評価軸を拡張・絞り込みながら進んだ：
`wasmCloud/Rust` → `Orleans・Pekko・Dapr・Ray等への代替調査` → `組み込み制約の追加` → `WAMR中心の現行構成`

| 候補 | 評価概要 | 現状の位置づけ |
|---|---|---|
| **wasmCloud** | NATSラティス上でwRPC通信、配置の自律性思想に近い。Kubernetes Operatorあり | 通信モデルはNATS/wRPC前提のため、Zenoh中心の現行方針とは別路線。参考アーキテクチャとして保持 |
| **Microsoft Orleans** | 仮想アクター（grain）。Activation Rebalancing/Repartitioningが負荷・通信頻度に基づく自律配置を実現し、コンセプトに最も近い | コンテナ/クラウド層の配置ロジック検討時の参考に限定 |
| **Apache Pekko** | クラスタシャーディング・CRDT分散データ | コンテナ層に限定して採用候補 |
| **Dapr** | sidecarとしてのvirtual actors/state store | コンテナ層のsidecarとしてなら採用可。組み込み機器には載せられない |
| **Ray** | リソース意識のあるactor配置 | コンテナ/クラウド層限定 |
| **Erlang/OTP・AtomVM** | 自己修復・関数型思想がmycflowと一致し、MCUまで統一できる唯一の選択肢として有力視された時期あり | **後退・除外**。OTPの監視機能がランタイム内蔵のため「監視をコンテナ層に分離する」責務分離方針と矛盾 |
| **NATS** | エッジ〜サーバでは強力 | MCU側では重すぎるためZenohに置き換え |

## 7. 全体構成イメージ（Docker初期段階）

```
┌─────────────────────────────────────┐
│        配置マネージャー（コンテナ層）   │
│  containerd + runwasi / Kubernetes   │
│  (スケジューリング + 監視 + ACL生成)   │
└──────────┬──────────────────────────┘
           │ etcd/Consul（配置メタデータ・トポロジグラフ）
┌──────────▼──────────────────────────┐
│  Node A          Node B    Node C   │
│  [WASMモジュール] [WASMモジュール] ...│
│  (WAMR / Zephyr・ESP-IDF上で実行のみ) │
└─────────────────────────────────────┘
   モジュール間通信: Zenoh（ACL制御）
   モジュール形式: WASM Component Model (WIT)
   配布: wkg + OCIレジストリ
   業務データ: CRDT / イベントソーシング
```

## 8. 未決定事項・今後の検討課題

- WASMコンポーネントモデル（WITインターフェース）の具体的設計
- モジュール間通信の同期・非同期モデルの最終決定
- 配置マネージャー自体の冗長化設計（単一障害点の回避）
- モジュールをステートレスに保ち、状態を外部ストアに委譲してマイグレーションコストを削減する方針の詳細化
- データフロートポロジが固定DAGか動的グラフかの決定（ACL生成タイミング・整合性保証に影響）
